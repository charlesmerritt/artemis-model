The repository's own documentation identifies exactly one runnable-today deliverable:

> "This is the only notebook in `notebooks/` that needs neither the `/mnt/d` drive nor Earth Engine, so it is the most runnable today." — `notes/treemap-cog-county-summary.md`
>
> `TreeMap_COG_County_Summary.ipynb` … "**Yes** (only one needing neither drive nor GEE)" — `notes/notebooks.md`

Its output — windowed per-county zonal statistics of a remote TreeMap-like raster — is a genuine tabular deliverable produced entirely by the repo's own code, and it exercises the same per-acre / area-expansion characterization of TreeMap pixels that ARTEMIS depends on (see `notes/treemap-methodology.md`). It is the best available representative of the project's primary purpose that can actually be reproduced here.

## Exact code that produced it

The artifact was generated by executing the repository's real notebook (unmodified logic) via `papermill`, changing **one parameter only** — the county-boundary source (see the substitution note below). Equivalent to running the notebook end-to-end from Jupyter as documented in `notebooks/README.md`.

```bash
# Environment (see "Dependencies" below)
papermill notebooks/TreeMap_COG_County_Summary.ipynb executed.ipynb \
  --cwd . \
  -p COUNTY_BOUNDARIES_URL <local census-schema counties file>
```

All other parameters kept at their committed defaults:

- `COG_OR_STAC_URL` = the notebook's default TreeMap-like COG on
  `data-to-science-opendata-bucket.s3.us-east-2.amazonaws.com`
  (`GTiff 154179×97279 float32 EPSG:5070`, 30 m grid, CONUS extent, `nodata=3.40282e+38`).
- `SUMMARY_MODE = "continuous"` (matches the float32 COG).
- `SOUTHEAST_STATE_FIPS` = the 11-state default.
- `MAX_FEATURES_FOR_DEMO = None` (all counties, not the demo subset).

The notebook opens the COG remotely and reads **one bounding-box window per county** (never the full raster), then computes continuous zonal statistics inside each county polygon. Runtime: ~4 minutes for all 1,009 counties.

## Environment deviations (honest notes)

Two environment-specific substitutions were required; **neither changes the notebook's computation, only its inputs' delivery**:

1. **Python 3.13 instead of 3.14.** `pyproject.toml` requires Python `>=3.14`. In this cloud environment `uv` cannot install CPython 3.14 because the `python-build-standalone` release lives on `github.com`, which is blocked by the egress policy (HTTP 403). The notebook's code is compatible with 3.13, and all pinned dependencies installed and ran cleanly on 3.13.

2. **County boundaries from a reachable mirror instead of `www2.census.gov`.** The notebook's default `load_vector()` downloads Census TIGER county boundaries from `https://www2.census.gov/...`, which is also egress-blocked here (HTTP 403) — even though `notes/notebooks.md` recorded it as reachable on 2026-07-14. To keep the *exact same code path and output schema*, the county polygons were supplied via the notebook's own `COUNTY_BOUNDARIES_URL` parameter, pointing at a local file built from Census-derived US county geometries (`plotly/datasets` mirror on `raw.githubusercontent.com`, which is reachable) reshaped to the census column schema the notebook expects (`STATEFP`, `COUNTYFP`, `NAME`, `GEOID`). This is the same underlying Census county data, obtained from a different host. The COG raster itself is the notebook's real default and was read live over the network.

## How to regenerate

On a machine with the `/mnt/d`-independent network path available (and, ideally, Python 3.14 via `uv`):

```bash
uv sync                       # or: pip install geopandas rasterio pandas numpy shapely pyogrio requests papermill ipykernel
uv run jupyter lab            # open notebooks/TreeMap_COG_County_Summary.ipynb and Run All
```

Leaving all parameters at their defaults reproduces these CSVs under
`data/interim/treemap_county_summary/`. If `www2.census.gov` is reachable in your
environment, the notebook downloads the county boundaries itself and no parameter
change is needed; otherwise set `COUNTY_BOUNDARIES_URL` to any local/remote county
vector carrying `STATEFP`/`COUNTYFP`/`NAME`/`GEOID` columns, as was done here.
